The Animation Is a Replay

July 16, 2026 · ~2,400 words · 4 live figures and a quiz, identical to the collaborative version · reading this as an AI agent? agents.md is for you

This is the version in my own voice. The collaboratively drafted version came first; same figures, same bench, my hand on the prose here.

Here is the rule this page exists to prove: run the simulation flat out, keep the event log, and treat every animation as a replay of that log. Get that separation right and the engine stops mattering - which I can demonstrate, because this page runs the same factory on four different engines: a TypeScript kernel I wrote by hand, a Rust kernel compiled to WebAssembly, SimScript, and the actual SimPy on an actual CPython, in your browser. One schema out of all four, one scrubber replaying all four, one throughput they all agree on.

I also owe a correction. Last post I wrote that discrete-event simulation was "the one piece of the stack with no drop-in browser library." That sentence lasted one day. One research pass turned up four browser-capable DES libraries - SimScript, discrete-sim, OESjs, SIM.JS - and Pyodide puts real SimPy in the browser besides. I put a "what's still open" section at the end of these posts precisely so they can be graded; this one got graded fast, so here is the grade, published.

The idea underneath survived the correction and came out stronger. Last time: the diagram is a projection of the model. This time, applied to motion: the animation is a projection of the event log. A simulation's clock is not your clock. It jumps event to event as fast as the CPU allows - 14 days in about 12 milliseconds - and if you chain the engine to the animation you get those sad simulations that trickle along in real time because someone wanted to watch. Keep the layers apart - model → engine → event log → presentation, nobody reaching past their neighbor - and one run replays at any speed, scrubs, diffs, and renders in any viewer that reads the schema.

I said there was no library. Here are four engines.

The whole bench hangs off one small contract. Every engine emits this; the stats, the hashes, and the replayer read nothing else:

{ "header": { "engine": "simpy-4.1.2", "seed": 42, "schema": 1, "params": { ... } },
  "events": [ { "t": 12.4, "type": "arrive", "id": 3 },
              { "t": 13.1, "type": "code_start", "id": 3 }, ... ] }

The model is the factory from last post, unchanged: arrivals every 25 minutes on average, three coding agents at 45 minutes a pass, CI at 10 minutes failing a quarter of the time, failures looping back as rework, one reviewer at 30 minutes. I can do this arithmetic on paper: review caps the system at 48 deploys/day. Twenty SimPy replications put the honest band at 47.9 ± 1.8 (3σ: 42.4 to 53.5). Paper prediction, statistical band - now run the engines into them.

Four real engines, one schema, nothing mocked. The TypeScript kernel lives in this page's source. The Rust kernel is 47 KB of WebAssembly (~53 KB with glue) out of this post's own crate. SimScript (~34 KB) comes from esm.sh when you first run it. SimPy is the real library on the real CPython, compiled to WebAssembly - and strictly opt-in: nothing Pyodide-touching is fetched until you press its button, which tells you the measured cost first (about 5 MB compressed, cached afterward). Rerun any row without touching anything and its hash comes back identical; no two engines ever share one. Determinism belongs to an engine; agreement between engines is statistics.

Every row inside the band, no hash repeated - and the second half of that sentence carries as much weight as the first. The four engines draw randomness four different ways (mulberry32, PCG32, SimScript's seeded streams, the Mersenne Twister), so their traces disagree on every event. What they are not allowed to disagree on is the statistics. Identical traces demand the same engine, same runtime, same seed - which is exactly what the hash chips prove, and I checked the strong version: this page's TS, Rust, SimScript, and SimPy runs reproduce the build machine's hashes bit for bit, Node to Chrome, native CPython to Pyodide.

Two clocks

Look at the wall-ms column again: 14 simulated days, 10 to 200 milliseconds. The virtual clock owes your wristwatch nothing. That asymmetry is the entire case for the event log - you never slow a simulation down to watch it; you run it flat out, keep the log, and pick a replay speed after the fact. The scrubber below will replay any row from Fig. 1. Four engines, one renderer. That is the thesis, moving.

Hand-built SVG, on purpose - presentation was the one layer where no library earned its keep. The replayer has never heard of any engine: it folds the event log into queues, busy slots, and a deploy count, which is why a Rust trace and a Python trace look identical here. Scrub as hard as you like; it rebuilds from t=0 on every seek, and at 5,000 events that costs nothing worth engineering around.

You could write the kernel in an afternoon

The kernel under all of this is almost insultingly small: a priority queue ordered by (time, sequence), and a loop that pops. Here it is twice - the TypeScript running Fig. 1's inline row and the Rust that became the 47 KB wasm row. Press either toy button: five events, two at the same instant, popped in order - b1 beats b2 because of the sequence number, not luck. I have chased enough flaky behavior to promise you that tie-break is where deterministic engines are won and lost.

Fig. 3 · The kernel, in both languages
TypeScript - the kernel behind Fig. 1's inline row
function makeKernel() {
  const heap = [];
  let seq = 0;
  const less = (a, b) =>
    a.t < b.t ||
    (a.t === b.t && a.seq < b.seq);
  return {
    schedule(t, event) {
      heap.push({ t, seq: seq++, event });
      // sift up
    },
    next() {
      // pop min by (t, seq), sift down
    }
  };
}
// The loop:
// while ((n = k.next()) && n.t <= horizon)
//   handle(n.event) may k.schedule(...)

Rust - the same kernel behind the wasm row
struct Scheduled<E> { t: f64, seq: u64, event: E }

impl<E> Ord for Scheduled<E> {
  fn cmp(&self, o: &Self) -> Ordering {
    o.t.total_cmp(&self.t)          // earliest first
     .then_with(|| o.seq.cmp(&self.seq))
  }
}

struct Kernel<E> {
  heap: BinaryHeap<Scheduled<E>>,
  seq: u64,
}
impl<E> Kernel<E> {
  fn schedule(&mut self, t: f64, event: E) {
    self.heap.push(Scheduled {
      t, seq: self.seq, event });
    self.seq += 1;
  }
  fn next(&mut self) -> Option<(f64, E)> {
    self.heap.pop().map(|s| (s.t, s.event))
  }
}

Full sources: the Rust crate ships with the post; the TypeScript is in this page. The kernel is the easy 10%, and the afternoon does not buy the rest: resource queues, cancellation, seeded streams, distributions, warm-up, replications, confidence intervals, serialization. That list is why SimPy exists. Adopt a library for semantics; keep the hand-rolled kernel for the day you need to own the core.

What I'll build


            
I built all four so this choice could run on data instead of vibes. What the bench told me: my TypeScript kernel took the longest to trust, because the queue-and-rework logic is exactly where the subtle bugs live and every line of it was mine to get wrong. The Rust port was mechanical once the semantics existed - 47 KB, fastest row on the board. SimScript needed the least adapter code and brought the most opinions. SimPy read closest to the problem statement - the factory is 40 lines of Python that sound like the prose that specified it - and drags the whole scientific ecosystem behind it. So:


            
    
                
  1. SimPy owns simulation semantics for the software factory, behind a narrow SimulationRuntime interface (dispatch / step / runUntil / reset(seed)) so no framework becomes the domain model.
    2. 
                
  2. The model stays declarative - the same canonical JSON/YAML discipline as the last post; engines consume it, none of them owns it.
    3. 
                
  3. The event-log schema is the contract. Every run ships with seed and params in the header; every viewer, stat, and diff reads logs, never engine state.
    4. 
                
  4. The Rust core is the eventual embedded engine, gated by conformance. When the factory needs simulation in the browser or at the edge, the port must reproduce SimPy's statistics against these reference logs before it is trusted - the same conformance-corpus pattern that guards my CrMS charting engine across languages. Fig. 1 is that corpus's first four entries.
    5. 
                
  5. Replications before beliefs. Single runs demo; 20-replication bands decide. The band in Fig. 1 came from SimPy in 2 seconds of CPU.

The quiz at the door


            
Same rule as last time, which is Geoffrey Litt's rule for his agents before it was mine: no merge until you can pass the quiz. Five questions.


            

                

                    Fig. 5 · Pass the quiz before you merge
                

                

                    

                        
1. Why not stream the engine's events to the browser live, as the simulation produces them?

                        

                            
                            
                            
                        

                        
A DES clock jumps event-to-event at CPU speed. Record the log, then replay it at a human speed - 1 min/s or 1 day/s, the log doesn't care.

                    

                    

                        
2. Two different engines, same scenario, same seed. Should their event-log hashes match? Should their deploys/day?

                        

                            
                            
                            
                        

                        
Determinism is a per-engine property: engine + runtime + seed → identical trace. Across engines the invariant is the statistics - Fig. 1's four rows agree on throughput and share no hash.

                    

                    

                        
3. Which layer owns simulation state during a replay?

                        

                            
                            
                            
                        

                        
The engine finished milliseconds after it started, and the renderer is stateless by design - Fig. 2 rebuilds the world from the log on every scrub. The log is the artifact.

                    

                    

                        
4. When does running SimPy in the browser (Pyodide) beat running it behind an API?

                        

                            
                            
                            
                        

                        
Pyodide is real CPython in WebAssembly: perfect for a static, self-contained page like this one, wrong for big runs, private models, or shared storage - that's when the FastAPI-plus-workers shape wins.

                    

                    

                        
5. All four engines land near the predicted 48/day. What result would have falsified the model instead?

                        

                            
                            
                            
                        

                        
Hashes must differ and wall clocks may differ - neither is evidence. A throughput outside the 3σ band would mean the semantics of some engine's model diverged: a real bug, findable precisely because the band exists.

                    

                    
0 of 5 answered.

                

            


            
What's still open


            
Three things I have not solved. The stat people will quote from this page is a single run, and the honest number is a replication band - replications-by-default is the fix, the SimPy row could do it, and the page does not yet. The bench proves agreement at one point in parameter space; a conformance corpus worth the name sweeps it (the p = 0.5 constraint-flip is verified in my build logs and still is not a figure). And a blog post that will download a Python interpreter rather than fake a demonstration is a strange object. I think it is the right kind of strange, and 5 MB came in under my fears, but it is still 5 MB nobody on a metered plan asked for - hence a button that quotes the price and fetches nothing until pressed.
