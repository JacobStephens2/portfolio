The Diagram Is Not the Model

July 15, 2026 · ~1,900 words · 3 live figures and a quiz

The most useful thing I learned researching diagramming tools for the software factory is a rule: the diagram is never the source of truth. It is a projection of a model that lives somewhere else - a YAML file, a state machine, an event log - and almost every good tooling decision falls out of that rule.

I put one question to a Perplexity model council - GPT-5.6 Sol, Claude Opus 4.8, and Grok 4.5 Low: what should AI agents use to illustrate, and eventually animate, the processes of a software factory? The three answers converged on four ideas. Agents should write diagrams as text. The text should be generated from a canonical model. Animation should encode state, not decorate. And the questions I actually care about - where does work pile up? - need simulation, not drawing.

A four-bullet summary is exactly the kind of thing I would read once, agree with, and forget. Geoffrey Litt's talk "Understanding: the New Bottleneck" is about that failure mode: as agents write more of the code, the scarce resource stops being production and becomes human understanding - which you need both to verify the work and to participate in improving it. Margaret Storey calls the gap cognitive debt. Andy Matuschak's essay "Why Books Don't Work" names the mechanism: reading produces the feeling of understanding without the retention. Litt's prescription is to steal from education - intuition before detail, interactive figures, micro-worlds, quizzes.

So this post is the experiment. Each of the four ideas is taught by the technique that fits it: a diff you can apply, a model you can edit, a factory you can run, and a quiz at the door.

The agent writes text; the browser draws it

All three models gave the same first answer: diagram-as-code. LLMs are far better at emitting structured text than at dragging shapes or generating pixel-accurate images, and text is what Git, PRs, and CI already know how to handle. Mermaid is the consensus default - it renders natively on GitHub and in any browser - with D2 as the upgrade when Mermaid's layout gets cramped, and Graphviz DOT for heavily branching trees.

The argument is easiest to feel as a code review. Below is a five-station release process and a proposal: work that fails CI should loop back to the coding agent instead of dying. Render the change.

Litt's list of explanation techniques includes the literate diff - explanation and artifact as one thing. A Mermaid block in a PR body is the cheapest possible version of that, and agents can emit it today.

The diagram is a projection, and projections can't drift

The strongest single idea came from GPT-5.6 Sol: make the canonical process definition a versioned data structure - JSON, YAML, TypeScript objects - and generate everything else from it. Mermaid for the docs. React Flow for the interactive view. XState machines for executable lifecycles. Simulation inputs for capacity planning. Diagrams become projections of the model, the way a database view is a projection of tables.

The payoff: projections cannot drift. A hand-maintained diagram is a second copy of the truth, and second copies rot - my own architecture diagrams run reliably one refactor behind the system they describe. Below, one JSON model drives two renderers that know nothing about each other: a flowchart and a station table. Animation is state, not decoration


            
On animation the three models split by target - Claude Opus 4.8 suggested D3 with GSAP for interactive views and Manim for rendered explainer video, the other two favored React Flow's animated edges driven by live events - but they agreed on the principle: animation should encode state or time, never garnish. Pulse an edge when an artifact moves. Color a node by running, blocked, failed. Replay yesterday's incident by scrubbing the event log. GPT-5.6 Sol framed it as three modes of one viewer: design (edit the intended process), live (watch current runs), replay (scrub history).


            
The animation worth building first is the token simulation: every work item is a dot, dots queue at stations, and capacity limits how many move at once. Which brings us to the last idea, because a token simulation is not really a diagram at all.


            
You can't see a bottleneck in a flowchart


            
Real factories are not planned with flowcharts. They are planned with discrete-event simulation - Siemens Plant Simulation, FlexSim, AnyLogic - because the questions that matter are ones a drawing cannot answer. Where will work accumulate? Which station is the constraint? Does adding capacity here change throughput at all? All three council models pointed at this discipline as the real home of "factory planning," and the shared advice was to borrow its concepts - queues, capacity, failure, rework - rather than buy the industrial suites.


            
This is also where the education thread is bluntest. Seymour Papert's Mindstorms argues you learn math by living in "Mathland"; Litt's version is the micro-world: don't read about the system, inhabit a small one. Alan Kay's 1972 paper "A Personal Computer for Children of All Ages" is the same claim about computers in general - the point of the machine is to level its user up.


            
So here is a micro-world: a small software factory. Requests arrive about every 25 minutes. Coding agents average 45 minutes per attempt. CI takes 10 minutes and fails at an adjustable rate; failures loop back as rework (hollow dots). A human reviewer averages 30 minutes per approval. Before you run it, commit to a prediction - White and Gunstone's predict-observe-explain, the oldest move in physics teaching.

A discrete-event simulation, the concept behind Siemens Plant Simulation and FlexSim, at toy scale. Dots are work items; hollow dots are rework. Exponential service times, Poisson arrivals. The meters show trailing utilization - watch which one pins at 100%.

            


            
The arithmetic behind what the sliders show: at a 25% failure rate each shipped item costs 1/(1 − 0.25) = 1.33 coding passes, about 60 agent-minutes, so 3 agents can push roughly 72 items a day - comfortably above the ~58 arriving. One reviewer at 30 minutes each caps the whole system at 48 a day. Review is the constraint. Doubling the agents moves nothing except where the pile sits; a second reviewer nearly doubles throughput. That is Goldratt's theory of constraints, and the sim lets you feel it in about thirty seconds - the reviewer's meter pins at 100% while the agents idle. Then push CI failure to 50% and watch the constraint migrate: each item now costs two coding passes, and the agents' meter is the one that saturates.

The quiz at the door


            
The sharpest line in my notes from Litt's talk is a rule he gives his own agents: don't send a PR unless I can pass a quiz on it - understanding as the regulator on speed. Matuschak's point applies squarely to this post: if you scrolled and nodded, you retained roughly nothing. Five questions.


            

                

                    Fig. 4 · Pass the quiz before you merge
                

                

                    

                        
1. An agent proposes a change to your release process, and teammates will review it in a PR. What should the agent produce?

                        

                            
                            
                            
                        

                        
Text diffs. GitHub renders Mermaid natively, so the PR shows the one-line change and the regenerated picture. An image can only be re-exported and eyeballed - and a generative model won't even draw the same boxes twice.

                    

                    

                        
2. Six months in, the architecture diagram no longer matches production. Which fix kills the whole class of problem?

                        

                            
                            
                            
                        

                        
Owners and reviews slow the rot; they don't stop it. A projection regenerated in CI cannot drift, because there is nothing to hand-edit.

                    

                    

                        
3. In Fig. 3, review is saturated at one reviewer. You double the coding agents from 3 to 6. Deploys per day go…

                        

                            
                            
                            
                        

                        
Goldratt: throughput is set by the constraint and nothing else. Review caps the system near 48 a day regardless of how fast work arrives at its queue.

                    

                    

                        
4. Which of these questions needs the simulation - the flowchart cannot answer it?

                        

                            
                            
                            
                        

                        
Structure and policy live in the drawing. Percentile latency emerges from variability, capacity, and rework interacting over time - you get it by running the process, in production or in simulation.

                    

                    

                        
5. Raising CI failure from 0% to 50% multiplies the coding work per shipped item by…

                        

                            
                            
                            
                        

                        
Expected attempts per shipped item is 1/(1 − p): at p = 0.5, two full coding passes each. Rework is a multiplier on whatever station sits upstream of the failing check.

What's still open


            
Litt flags his own caveat: interactive figures can be a crutch. Did you learn queueing behavior just now, or did you move sliders? The quiz is a partial check, and my honest answer for myself is "partly" - I trust the 1/(1 − p) arithmetic because I derived it, and I trust the bottleneck intuition more because I watched the meter pin. The other open question is whether hand-building these figures in vanilla SVG and JavaScript contradicts the post's own advice. I don't think it does - each figure is a small model with render functions over it, which is the thesis in miniature - but the real test is whether the factory's diagrams still match the factory six months from now, and that one only time can grade.
