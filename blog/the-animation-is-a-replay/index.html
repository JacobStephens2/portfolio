The Animation Is a Replay

July 16, 2026 · ~2,700 words · 6 figures and a quiz · reading this as an AI agent? agents.md is for you

Run the simulation at full speed, record an event log, and treat every animation as a replay of that log - do that, and the simulation engine itself becomes swappable. This page proves the rule the direct way: one factory model, four discrete-event engines - a hand-rolled TypeScript kernel, a Rust kernel compiled to WebAssembly, SimScript, and the real SimPy running on a real CPython in your browser - all emitting one event-log schema, all replayed by one scrubber, all landing on the same throughput.

It is also a correction. The last post closed by calling discrete-event simulation "the one piece of the stack with no drop-in browser library." That claim lasted one day. A research pass (sources at the bottom) turned up SimScript, discrete-sim, OESjs, and SIM.JS in the browser-capable column alone - and Pyodide puts actual SimPy there too. When a "what's still open" section gets graded this fast, the honest move is to publish the grade.

The deeper idea survived the correction and got stronger. The last post argued the diagram is a projection of the model. This one extends the rule to motion: the animation is a projection of the event log. A simulation's clock is not your clock - it advances as fast as the CPU allows, simulating 14 days in about 12 milliseconds. Confuse the two clocks and you get simulations shackled to their animations, playing out in awkward real-time. Separate them - model → engine → event log → presentation, each layer forbidden to reach past its neighbor - and the same run can be replayed at any speed, scrubbed, diffed, shared, and rendered by any viewer that reads the schema.

The event log is not debugging exhaust; it is the interface. The producer owns virtual time, the viewer owns presentation time, and neither needs a reference to the other.

I said there was no library. Here are four engines.

The contract that makes the bench work is one small schema. Every engine adapter emits it; the stats, the hashes, and the replayer read nothing else:

The model is the exact factory from the last post's simulation: requests arrive every 25 minutes on average, three coding agents take 45 minutes a pass, CI takes 10 minutes and fails 25% of the time, failures loop back as rework, one reviewer takes 30 minutes. Closed form says review caps the system at 48 deploys/day. The chart makes the arithmetic - and the point where it stops being true - visible.

Rework multiplies coding effort by 1/(1-p), so coding capacity is 96(1-p) deploys/day. At the default p = 0.25, coding can supply 72/day and review caps throughput at 48/day. At p = 0.50 the capacities are tied; only above 0.50 is coding strictly the constraint.

A 200-replication SimPy study puts 14-day runs at 47.3 ± 1.9 deploys/day (mean ± run-to-run standard deviation; 3σ: 41.6 to 53.0). Now run the engines against it.

Four real engines, one schema. The TypeScript kernel is inline in this page. The Rust kernel is a 47 KB WebAssembly module (~53 KB with its JS glue) built from this crate. SimScript (~34 KB) loads from esm.sh on first run. SimPy is the real library on a real CPython compiled to WebAssembly - strictly opt-in behind its button: nothing Pyodide-related is fetched until you click, and the measured cost is about 5 MB compressed (CPython 3.13, its stdlib, micropip, and the SimPy wheel), cached after the first run. Same engine + same seed reproduces the same hash; different engines never share one - determinism is per-engine, agreement across engines is statistical.

Every row lands inside the band; no two rows share a hash. That second part matters as much as the first. Each engine draws its randomness differently (mulberry32, PCG32, SimScript's seeded streams, Python's Mersenne Twister), so their traces differ event by event - what they must agree on is the statistics. If you want two runs with identical traces, you need the same engine, the same runtime, and the same seed: that is what the hash chip proves when you rerun a row without touching anything.

Two clocks

Notice the wall-ms column: 14 simulated days in 10-200 milliseconds. The engine's virtual clock jumps from event to event; it has no relationship to the wall clock on your wrist. That asymmetry is the whole argument for the event log. You do not slow the simulation down to watch it - you run it flat out, keep the log, and replay the log at whatever speed teaches. The replayer is hand-built SVG on purpose: presentation is the one layer where nothing needed a library. It knows nothing about any engine - it reconstructs queues, busy slots, and the deploy count purely from the event log, which is why a Rust trace and a Python trace render identically. Scrub freely; the reconstruction is recomputed from t=0 on every seek, and 5,000 events make that free.

You could write the kernel in an afternoon

The engine at the heart of all this is embarrassingly small: a priority queue ordered by (time, sequence) and a loop that pops the next event. Here it is twice - the TypeScript that runs the inline row of Fig. 3, and the Rust that compiled to the 47 KB wasm row. The toy button under each schedules five events, two at the same instant, and pops them - watch b1 beat b2 on sequence, not luck. That tie-break is the difference between a deterministic engine and a flaky one.

Fig. 5 · The kernel, in both languages
TypeScript - the kernel behind Fig. 3's inline row
function makeKernel() {
  const heap = [];
  let seq = 0;
  const less = (a, b) =>
    a.t < b.t ||
    (a.t === b.t && a.seq < b.seq);
  return {
    schedule(t, event) {
      heap.push({ t, seq: seq++, event });
      // sift up
    },
    next() {
      // pop min by (t, seq), sift down
    }
  };
}
// The loop:
// while ((n = k.next()) && n.t < horizon)
//   handle(n.event) may k.schedule(...)

Rust - the same kernel behind the wasm row
struct Scheduled<E> { t: f64, seq: u64, event: E }

impl<E> Ord for Scheduled<E> {
  fn cmp(&self, o: &Self) -> Ordering {
    o.t.total_cmp(&self.t)          // earliest first
     .then_with(|| o.seq.cmp(&self.seq))
  }
}

struct Kernel<E> {
  heap: BinaryHeap<Scheduled<E>>,
  seq: u64,
}
impl<E> Kernel<E> {
  fn schedule(&mut self, t: f64, event: E) {
    self.heap.push(Scheduled {
      t, seq: self.seq, event });
    self.seq += 1;
  }
  fn next(&mut self) -> Option<(f64, E)> {
    self.heap.pop().map(|s| (s.t, s.event))
  }
}

Full sources: the Rust crate ships with this post; the TypeScript is in this page's source. The kernel is the easy 10%. What the afternoon does not buy you: resource queues, cancellation, seeded random streams, distribution sampling, warm-up periods, batch replications, confidence intervals, serialization. Those are the reasons SimPy and friends exist - adopt them for semantics, and save the hand-rolled kernel for when you need to own the core.

What I'll build


            
The point of building all four was to choose with data instead of vibes. What the bench measured: the TypeScript kernel took the longest to trust (the queue-and-rework logic is exactly where subtle bugs live, and I wrote every line of it); the Rust port was mechanical once the semantics existed, weighs 47 KB, and is the fastest row; SimScript was the least adapter code but brings its own modeling idioms; SimPy read the closest to the problem statement - the factory is 40 lines of Python that look like the prose description - and carries the ecosystem. So:


            
    
                
  1. SimPy owns simulation semantics for the software factory, behind a narrow SimulationRuntime interface (dispatch / step / runUntil / reset(seed)) so no framework becomes the domain model.
    2. 
                
  2. The model stays declarative - the same canonical JSON/YAML discipline as the last post; engines consume it, none of them owns it.
    3. 
                
  3. The event-log schema is the contract. Every run ships with seed and params in the header; every viewer, stat, and diff reads logs, never engine state.
    4. 
                
  4. The Rust core is the eventual embedded engine, gated by conformance. When the factory needs simulation in the browser or at the edge, the port must reproduce SimPy's statistics against these reference logs before it is trusted - the same conformance-corpus pattern that guards my CrMS charting engine across languages. Fig. 3 is that corpus's first four entries.
    5. 
                
  5. Replications before beliefs. Single runs demo; replicated distributions decide. The 200-run band in Fig. 3 came from SimPy in about 20 seconds of CPU.

The quiz at the door


            
Same rule as last time - Geoffrey Litt's rule, borrowed from his agents: no merge until you can pass the quiz. Five questions.


            

                

                    Quiz · Pass before you merge
                

                

                    

                        
1. Why not stream the engine's events to the browser live, as the simulation produces them?

                        

                            
                            
                            
                        

                        
A DES clock jumps event-to-event at CPU speed. Record the log, then replay it at a human speed - 1 min/s or 1 day/s, the log doesn't care.

                    

                    

                        
2. Two different engines, same scenario, same seed. Should their event-log hashes match? Should their deploys/day?

                        

                            
                            
                            
                        

                        
Determinism is a per-engine property: engine + runtime + seed → identical trace. Across engines the invariant is the statistics - Fig. 3's four rows agree on throughput and share no hash.

                    

                    

                        
3. Which layer owns simulation state during a replay?

                        

                            
                            
                            
                        

                        
The engine finished milliseconds after it started, and the renderer is stateless by design - Fig. 4 rebuilds the world from the log on every scrub. The log is the artifact.

                    

                    

                        
4. When does running SimPy in the browser (Pyodide) beat running it behind an API?

                        

                            
                            
                            
                        

                        
Pyodide is real CPython in WebAssembly: perfect for a static, self-contained page like this one, wrong for big runs, private models, or shared storage - that's when the FastAPI-plus-workers shape wins.

                    

                    

                        
5. All four engines land near the predicted 48/day. What result would have falsified the model instead?

                        

                            
                            
                            
                        

                        
Hashes must differ and wall clocks may differ - neither is evidence. A throughput outside the 3σ band would mean the semantics of some engine's model diverged: a real bug, findable precisely because the band exists.

                    

                    
0 of 5 answered.

                

            


            
What's still open


            
Three honest items. This page shows single runs next to a replicated distribution, but the stat everyone will quote is still the single run - the right fix is replications-by-default, which the SimPy row could do and the page does not yet. Second, Fig. 2 shows the analytical parameter sweep, while the four-engine bench proves agreement only at p = 0.25; a real conformance corpus runs every adapter across the sweep. Third, a page that downloads a Python interpreter rather than fake a demo is a strange object; I think the strangeness is the good kind, and the measured 5 MB is smaller than I feared, but it is still 5 MB someone on a metered connection did not ask for - which is why that button states the cost and nothing loads until it is pressed.
